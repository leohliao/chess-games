    <article class="markdown-body entry-content" itemprop="text"><h1><a id="user-content-chess" class="anchor" href="#chess" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Chess</h1>
<p>Write a <a href="http://en.wikipedia.org/wiki/Chess">chess game</a> using object-oriented programming.</p>
<p><strong>Please read all through the various phases before proceeding.</strong></p>
<p>Review the <a href="/appacademy/curriculum/blob/master/ruby/assets/Chess_Diagram.png">Chess UML Diagram</a> to get an overview of what you'll be creating. The diagram is beneficial for getting an idea of how different aspects of the project fit together; however, you should <strong>code your project by following the instructions closely and using the diagram only as a reference</strong>. It'll be helpful for clearing up any confusion about how  classes inherit from or relate  to one another.</p>
<p>You must split your program into multiple files. Use <a href="http://www.ruby-doc.org/core-2.1.2/Kernel.html#method-i-require_relative"><code>require_relative</code></a> to load files. Make separate files for each class. Give files <code>snake_case</code> titles.</p>
<p>Don't forget to keep using Git. Look back at <a href="/appacademy/curriculum/blob/master/ruby/projects/minesweeper">Thursday's instructions</a> and read the Git portions again if you need a reminder. Committing and branching are important habits for any kind of software developer.</p>
<h2><a id="user-content-learning-goals" class="anchor" href="#learning-goals" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Learning Goals</h2>
<ul>
<li>Know when and why private methods are used</li>
<li>Be able to read UML and understand the benefits of UML</li>
<li>Be familiar with how to use the Singleton module</li>
<li>Know how to use modules</li>
<li>Know how class inheritance works</li>
</ul>
<h2><a id="user-content-phase-i-board" class="anchor" href="#phase-i-board" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase I: <code>Board</code></h2>
<p>Your <code>Board</code> class should hold a 2-dimensional array (an array of arrays). Each position in the board either holds a moving <code>Piece</code> or a <code>NullPiece</code> (<code>NullPiece</code> will inherit from <code>Piece</code>).</p>
<p>You'll want to create an empty <code>Piece</code> class as a placeholder for now. Write code for <code>#initialize</code> so we can setup the board with instances of <code>Piece</code> in locations where a <code>Queen</code>/<code>Rook</code>/<code>Knight</code>/ etc. will start and <code>nil</code> where the <code>NullPiece</code> will start.</p>
<p>The <code>Board</code> class should have a <code>#move_piece(start_pos, end_pos)</code> method. This should update the 2D grid and also the moved piece's position. You'll want to raise an exception if:</p>
<ol start="0">
<li>there is no piece at <code>start_pos</code> or</li>
<li>the piece cannot move to <code>end_pos</code>.</li>
</ol>
<p><strong>Time to test!</strong> Open up pry and <code>load 'board.rb'</code>. Create an instance of <code>Board</code> and check out different positions with <code>board[pos]</code>. Do you get back <code>Piece</code> instances where you expect to? Test out <code>Board#move_piece(start_pos, end_pos)</code>, does it raise an error when there is no piece at the start? Does it successfully update the <code>Board</code>?</p>
<h2><a id="user-content-phase-ii-display" class="anchor" href="#phase-ii-display" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase II: <code>Display</code></h2>
<p>Write a <code>Display</code> class to handle your rendering logic. Your <code>Display</code> class should access the board. Require the <a href="https://github.com/fazibear/colorize"><code>colorize</code></a> gem so you can render in color.</p>
<p>Download this <code>cursor.rb</code> <a href="/appacademy/curriculum/blob/master/ruby/projects/chess/cursor.rb?raw=true">file</a>. An instance of <code>Cursor</code> initializes with a <code>cursor_pos</code> and an instance of <code>Board</code>. The cursor manages user input, according to which it updates its <code>@cursor_pos</code>. The display will render the square at <code>@cursor_pos</code> in a different color. Within <code>display.rb</code> require <code>cursor.rb</code> and set the instance variable <code>@cursor</code> to <code>Cursor.new([0,0], board)</code>.</p>
<p>In <code>cursor.rb</code> we've provided a <code>KEYMAP</code> that translates keypresses into actions and movements. The <code>MOVES</code> hash maps possible movement differentials. You can use the <code>#get_input</code> method as is. <code>#read_char</code> handles console input. Skim over <code>#read_char</code> to gain a general understanding of how the method works. It's all right if the <code>STDIN</code> methods are unfamiliar. Don't fret the details.</p>
<p>Fill in the <code>#handle_key(key)</code> method. Use a <a href="http://ruby-doc.org/docs/keywords/1.9/Object.html#method-i-case">case statement</a> that switches on the value of <code>key</code>. Depending on the <code>key</code>, <code>#handle_key(key)</code> will a) return the <code>@cursor_pos</code> (in case of <code>:return</code> or <code>:space</code>), b) call <code>#update_pos</code> with the appropriate movement difference from <code>MOVES</code> and return <code>nil</code> (in case of <code>:left</code>, <code>:right</code>, <code>:up</code>, and <code>:down</code>), or c) exit from the terminal process (in case of <code>:ctrl_c</code>). Later we will use our <code>Player</code> and <code>Game</code> classes to handle the movement of pieces.</p>
<p><strong>NB:</strong> To exit a terminal process, use the <code>Process.exit</code> method. Pass it the status code <code>0</code> as an argument. You can read more about <code>exit</code> <a href="http://ruby-doc.org/core-2.2.0/Process.html#method-c-exit">here</a>.</p>
<p>Fill in the <code>#update_pos(diff)</code> method. It should use the <code>diff</code> to reassign <code>@cursor_pos</code> to a new position. You may wish to write a <code>Board#in_bounds</code> method to ensure you update <code>@cursor_pos</code> only when the new position is on the board.</p>
<p>Render the square at the <code>@cursor_pos</code> display in a different color. Test that you can move your cursor around the board by creating and calling a method that loops through <code>Display#render</code> and <code>Cursor#get_input</code> (much as <code>Player#play_turn</code> will function later!).</p>
<p>A nice but optional addition to your cursor class is a boolean instance variable <code>selected</code> that will allow you to display the cursor in a different color when it has selected a piece. To implement this you will need to <code>#toggle_selected</code> everytime <code>:return</code> or <code>:space</code> is hit.</p>
<p><strong>NB:</strong> <strong>If you're stuck on making a cursor for more than <em>30 minutes</em>, please call for help from a TA</strong>. Fancy cursors are cool, but the purpose of today is to become more familiar with Object-oriented Programming.</p>
<p><strong>Time to test!</strong> This time you should run <code>ruby display.rb</code>. Does your board render as you would expect? Make sure that as you move your cursor the display updates accordingly. Test the cursor's behavior when you try and move it off the board (the edge cases if you will). Does it do what you expect?</p>
<p><strong>Code Review Time:</strong> Before moving on to piece logic, get a code review from a TA!</p>
<h2><a id="user-content-phase-iii-pieces" class="anchor" href="#phase-iii-pieces" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase III: <code>Pieces</code></h2>
<p>There are many different kinds of pieces in chess, and each moves a specific way.  Based on their moves, they can be placed in four categories:</p>
<ol start="0">
<li>Sliding pieces (Bishop/Rook/Queen)</li>
<li>Stepping pieces (Knight/King)</li>
<li>Null pieces (occupy the 'empty' spaces)</li>
<li>Pawns (we'll do this class last)</li>
</ol>
<p>Start by writing a <code>Piece</code> parent class that contains the functionality common to all pieces. A key method of <code>Piece</code> is <code>#moves</code>, which should return an array of places a <code>Piece</code> can move to. Of course, every piece will move differently, so you can't write (<strong>implement</strong>) the <code>#moves</code> method of <code>Piece</code> without subclasses.</p>
<p>You should make <a href="/appacademy/curriculum/blob/master/ruby/readings/modules.md">modules</a> for <code>SlidingPiece</code> and <code>SteppingPiece</code>. The <code>SlidingPiece</code> module can implement <code>#moves</code>, but it needs to know what directions a piece can move in (diagonal, horizontally/vertically, both). Classes that <code>include</code> the module <code>SlidingPiece</code> (<code>Bishop</code>/<code>Rook</code>/<code>Queen</code>) will need to implement a method <code>#move_dirs</code>, which <code>#moves</code> will use.</p>
<p>Your <code>Piece</code> will need to (1) track its position and (2) hold a reference to the <code>Board</code>. Classes that <code>include SlidingPiece</code> in particular need the <code>Board</code> so they know to stop sliding when blocked by another piece. Don't allow a piece to move into a square already occupied by the same color piece, or to move a sliding piece past a piece that blocks it.</p>
<p>The <code>NullPiece</code> class should include <a href="http://ruby-doc.org/stdlib-1.9.3/libdoc/singleton/rdoc/Singleton.html">the <code>singleton</code> module</a>. It will not need a reference to the <code>Board</code> - in fact its <code>initialize</code> method should take no arguments. Make sure you have a way to read its <code>color</code> and <code>symbol</code>.</p>
<p><strong>After completing each piece</strong> load the file in pry and make sure it moves properly. <strong>Once you have completed all pieces</strong> refactor your <code>Board#initialize</code> so that all your pieces are placed in their respective starting positions. Run <code>ruby display.rb</code> to check that they show up in the proper locations.</p>
<p>For now, do not worry if a move leaves a player in check.</p>
<h2><a id="user-content-phase-iv-boardin_checkcolor-and-boardcheckmatecolor" class="anchor" href="#phase-iv-boardin_checkcolor-and-boardcheckmatecolor" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase IV: <code>Board#in_check?(color)</code> and <code>Board#checkmate?(color)</code></h2>
<p>The <code>Board</code> class should have a method <code>#in_check?(color)</code> that returns whether a player is in check. You can implement this by (1) finding the position of the King on the board then (2) seeing if any of the opposing pieces can move to that position.</p>
<p>Then write a <code>#checkmate?(color)</code> method. If the player is in check, and if none of the player's pieces have any <code>#valid_moves</code> (to be implemented in a moment), then the player is in checkmate.</p>
<h2><a id="user-content-phase-v-piecevalid_moves" class="anchor" href="#phase-v-piecevalid_moves" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase V: <code>Piece#valid_moves</code></h2>
<p>You will want a method on <code>Piece</code> that filters out the <code>#moves</code> of a <code>Piece</code> that would leave the player in check. A good approach is to write a <code>Piece#move_into_check?(end_pos)</code> method that will:</p>
<ol start="0">
<li>Duplicate the <code>Board</code> and perform the move</li>
<li>Look to see if the player is in check after the move (<code>Board#in_check?</code>)</li>
</ol>
<p>To do this, you'll have to write a <code>Board#dup</code> method. Your <code>#dup</code> method should duplicate not only the <code>Board</code>, but the pieces on the <code>Board</code>. <strong>Be aware</strong>: Ruby's <code>#dup</code> method does not call <code>dup</code> on the instance variables, so you may need to write your own <code>Board#dup</code> method that will <code>dup</code> the individual pieces as well.</p>
<h4><a id="user-content-a-note-on-deep-duping-your-board" class="anchor" href="#a-note-on-deep-duping-your-board" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>A note on deep duping your board</h4>
<p>As we saw in one of the <a href="/appacademy/curriculum/blob/master/ruby/projects/recursion">recursion exercises</a>, Ruby's native <code>#dup</code> method does not make a <strong>deep copy</strong>.  This means that nested arrays <strong>and any arrays stored in instance variables</strong> will not be copied by the normal <code>dup</code> method:</p>
<div class="highlight highlight-source-ruby"><pre><span class="pl-c"><span class="pl-c">#</span> Example: if piece position is stored as an array</span>
queen <span class="pl-k">=</span> <span class="pl-c1">Queen</span>.<span class="pl-k">new</span>([<span class="pl-c1">0</span>, <span class="pl-c1">1</span>])
queen_copy <span class="pl-k">=</span> queen.dup

<span class="pl-c"><span class="pl-c">#</span> shouldn't modify original queen</span>
queen_copy.position[<span class="pl-c1">0</span>] <span class="pl-k">=</span> <span class="pl-s"><span class="pl-pds">"</span>CHANGED<span class="pl-pds">"</span></span>
<span class="pl-c"><span class="pl-c">#</span> but it does</span>
queen.position <span class="pl-c"><span class="pl-c">#</span> =&gt; ["CHANGED", 1]</span></pre></div>
<h3><a id="user-content-caution-on-duping-pieces" class="anchor" href="#caution-on-duping-pieces" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Caution on duping pieces</h3>
<p>If your piece holds a reference to the original board, you will need to update this reference to the new duped board. Failure to do so will cause your duped board to generate incorrect moves!</p>
<h3><a id="user-content-an-alternative-to-duping" class="anchor" href="#an-alternative-to-duping" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>An alternative to duping?</h3>
<p>Another way to write <code>#valid_moves</code> would be to make the move on the original board, see if the player is in check, and then "undo" the move. However, this would require that <code>Board</code> have a method to undo moves.</p>
<p>Once you write your <code>Board#dup</code> method, it'll be very straightforward to write <code>Piece#valid_moves</code> without complicated do/undo logic.</p>
<p>It will likely help at this point to put some debug information into your <code>Display</code> class. Set a boolean variable on initialize of your <code>Display</code> for whether or not to show your debug info (you want to be able to easily turn this off). If that flag is set to true, then output some debug info, such as the selected piece's available moves, whether your opponent is in check, and so on.</p>
<p><strong>Test each piece's <code>#valid moves</code>!</strong> In pry <code>load 'board.rb'</code> and create a <code>Board</code> instance. Grab an instance of each type of piece (ie. from its position on the board) and check that calling <code>#valid_moves</code> returns what we expect. When you are satisfied it works <strong>call a TA over for a code review!</strong></p>
<h3><a id="user-content-further-board-improvements" class="anchor" href="#further-board-improvements" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Further <code>Board</code> improvements</h3>
<p>Modify your <code>Board#move_piece</code> method so that it only allows you to make valid moves. Because <code>Board#move_piece</code> needs to call <code>Piece#valid_moves</code>, <code>#valid_moves</code> must not call <code>Board#move_piece</code>. But <code>#valid_moves</code> needs to make a move on the duped board to see if a player is left in check. For this reason, write a method <code>Board#move_piece!(start_pos, end_pos)</code> which makes a move without checking if it is valid.</p>
<p><code>Board#move_piece</code> should raise an exception if it would leave you in check.</p>
<h2><a id="user-content-phase-vi-game" class="anchor" href="#phase-vi-game" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase VI: <code>Game</code></h2>
<p>Only when done with the basic Chess logic (moving, check, checkmate) should you begin writing user interaction code.</p>
<p>Write a <code>Game</code> class that constructs a <code>Board</code> object, then alternates between players (assume two human players for now) prompting them to move. The <code>Game</code> should handle exceptions from <code>Board#move_piece</code> and report them.</p>
<p>You should write a <code>HumanPlayer</code> class with one method (<code>#play_turn</code>). This method should call <code>Cursor#get_input</code> and appropriately handle the different responses (a cursor position or nil).  In that case, <code>Game#play</code> method just continuously calls <code>play_turn</code>.</p>
<p>It is not a requirement to write a <code>ComputerPlayer</code>, but you may do this as a bonus. If you write your <code>Game</code> class cleanly, it should be relatively straightforward to add new player types at a later date.</p>
<h2><a id="user-content-tips" class="anchor" href="#tips" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Tips</h2>
<ul>
<li>Do not implement tricky moves like "en passant". Don't implement castling, draws, or pawn promotion either. You <strong>should</strong> handle check and check mate, however.</li>
<li>Once you get some of your pieces moving around the board, <strong>call over your TA for a code-review</strong>.</li>
<li>Here's a four-move sequence to get to checkmate from a starting board for your checkmate testing:
<ul>
<li>f2, f3</li>
<li>e7, e5</li>
<li>g2, g4</li>
<li>d8, h4</li>
</ul>
</li>
</ul>
<h2><a id="user-content-phase-vi-bonus-round" class="anchor" href="#phase-vi-bonus-round" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Phase VI: Bonus Round!</h2>
<p>After completing each phase of the project, please remember to go back and make your code truly stellar, practicing all you know about coding style, encapsulation, and exception handling.</p>
<ul>
<li>DRY out your code</li>
<li>Use exception handling, and make sure to deal with bad user input</li>
<li>Method decomposition (pull chunks of code into helper methods)</li>
<li>Make helper methods private</li>
<li>Jazz up your User Interface (UI) with <a href="http://en.wikipedia.org/wiki/Chess_symbols_in_Unicode">unicode</a>.</li>
<li>Make a chess AI! Start with totally random moves. Next, capture pieces when possible. When you have this functionality working start giving your pieces some strategy! You can do it!</li>
</ul>
